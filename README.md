<h2>Technology Stack Decision Process</h2>

<h3>Javascript vs Typescript</h3>
<p><b>Typescript offers clear advantages, but the learning curve was a concern.</b></p>
<table>
  <tr>
    <th>Javascript</th>
    <th>Typescript</th>
  </tr>
  <tr>
    <td>- No need to learn type application separately; development can start immediately.</td>
    <td>- Static typing reduces debugging time, making it more efficient in terms of time resources. Type documentation is possible.</td>
  </tr>
  <tr>
    <td>- Type errors are caught only during runtime, making debugging difficult.</td>
    <td>- Learning curve.</td>
  </tr>
</table>

<h3>Final Decision: Typescript</h3>
<p>Given that type errors are caught during the development phase rather than at runtime, it was decided that overcoming the learning curve would be worthwhile. We agreed on using Typescript for its debugging efficiency.</p>

<h3>Type Documentation Conventions</h3>
<p><b>Interface and Type Alias Usage:</b></p>
<ul>
  <li>Interfaces are used for page-specific models and API data types, prefixed with "I".</li>
  <li>Type aliases are used for function components and styled-components props.</li>
</ul>

<p><b>Outcome:</b></p>
<ul>
  <li>Overall reduction in debugging time due to Typescript's static typing.</li>
  <li>Improved code readability and maintainability, especially in team environments where different members work on the same pages.</li>
</ul>

<h3>State Management Tool Selection</h3>

<h4>Advantages of Redux Toolkit</h4>
<p>Redux Toolkit reduces boilerplate code significantly, improving productivity in state management. It was concluded that using Redux Toolkit would save development time and simplify maintenance.</p>

<h4>Challenges with Redux Toolkit</h4>
<ul>
  <li>Asynchronous data state management was complex, requiring additional middleware (e.g., redux-thunk or redux-saga).</li>
  <li>Multiple actions and reducers for each state (e.g., request start, success, failure) added complexity.</li>
</ul>

<h4>Solution: React-Query</h4>
<p>React-Query was used for caching and managing asynchronous data states, while Redux Toolkit was kept for global state management. This improved cache management and eliminated unnecessary store management.</p>

<h4>Query Key Management</h4>
<p>During refactoring, query keys were centralized and made reusable by creating a common-utils-queryKey.ts file. This ensured that all queries using the same key shared the same cache.</p>

<h3>Outcome:</h3>
<ul>
  <li>Effective query key management led to better cache reuse and reduced network requests.</li>
  <li>Next time, we plan to set up query keys at the beginning of the project to improve efficiency.</li>
  <li>We plan to use React-Query DevTools in future projects to visualize cached data.</li>
</ul>

<h3>Testing Code Approach</h3>

<h4>Why Testing Was Needed</h4>
<ul>
  <li>For systematic bug fixes</li>
  <li>To build trust within the team</li>
  <li>To improve code quality</li>
</ul>

<h4>Concerns</h4>
<p>Since the frontend team was new to Jest and testing libraries, there was uncertainty about overcoming the learning curve within a limited timeframe.</p>

<h4>Decision: Unit Testing for Helper Functions</h4>
<p>The team decided to start with unit tests for pure functions and helper functions. These would help with input/output validation and avoid significant delays in testing.</p>

<h4>Outcome:</h4>
<ul>
  <li>Unit testing pure functions, such as the calendar generation function, was less difficult than expected.</li>
  <li>However, expanding the scope of testing to cover more components and interactions was difficult within the project timeframe.</li>
</ul>

<h4>Future Plan</h4>
<p>In future projects, we plan to expand the testing coverage by gradually including function components and integrations after establishing testing conventions early on.</p>

<h3>Collaboration Method</h3>
<h4>Daily Standups</h4>
<p>Team members held daily standups to share their progress and challenges. These provided a chance to discuss blockers and offered a platform for everyone to ask for help and input.</p>

<h4>Pair Programming</h4>
<p>Pair programming was used for some tasks to help overcome unfamiliar concepts or tricky bugs. This promoted knowledge sharing and quick problem resolution.</p>

<h4>Code Review</h4>
<p>Regular code reviews were held to ensure consistency and improve code quality. This encouraged team members to communicate better and share best practices.</p>

<h4>Outcome:</h4>
<ul>
  <li>These practices enhanced team collaboration and allowed for better distribution of tasks and expertise.</li>
  <li>As a result, the project saw smoother development flow despite the time constraints.</li>
</ul>
